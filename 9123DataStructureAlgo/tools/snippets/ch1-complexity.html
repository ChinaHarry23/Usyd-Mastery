Lecture 1 connects informal “how long does it take?” reasoning with asymptotic notation and with the basic operations model used on the homework slide (“Big O Examples”).

Why big-O?

Big-O describes how time or space requirements grow as input size grows — so we can compare algorithms and predict behaviour at scale, without tying the analysis to one machine or one constant factor.

Time: linear search vs binary search (“books in a pile”)

The slides use a pile of sorted books:

  • Linear search — scan from one end; in the worst case you inspect every book: \(O(n)\) time for \(n\) items.
  • Binary search — repeatedly discard half of the remaining range (data must be ordered): \(O(\log n)\) comparisons in the usual model.

Illustrative counts from the slides: for 10 books, binary search needs at most about \(\lceil \log_2 10\rceil = 4\) checks; for 100 books, about 7 — versus up to \(n\) for linear scan.

Space (extra memory)

For these search patterns, auxiliary space is often \(O(1)\): you only need a few indices or pointers beyond the input itself (the “note of where you are” in the pile analogy).

\(O\), \(\Omega\), and \(\Theta\)

  • \(O(\cdot)\) — asymptotic upper bound (worst-case growth, loosely “not faster than…” in the upper-bound sense).
  • \(\Omega(\cdot)\) — asymptotic lower bound.
  • \(\Theta(\cdot)\) — tight bound: same growth rate up to constant factors (sandwiched between matching upper and lower bounds).

Basic operations counted as \(O(1)\)

As on the homework slide, treat each of the following as a constant-time primitive in the RAM model: single arithmetic; single assignment; single Boolean test; single read/write of a primitive value.

Combining rules (homework patterns)

  • Sequential statements: costs add (dominant term wins).
  • Loop running \(N\) times with \(O(1)\) body: typically \(O(N)\).
  • Nested loops over \(N\) and \(M\) with \(O(1)\) inner body: typically \(O(NM)\); when both scale with \(n\), you often see \(O(n^2)\).
  • If one branch costs \(O(n)\) and the other \(O(1)\), the fragment is often \(O(n)\) in worst-case analysis.

Constants still matter in practice. Big-O hides leading constants: an \(O(n)\) algorithm with a huge constant can beat an \(O(n \log n)\) algorithm only for small \(n\). Use asymptotics for scalability; profile or measure when constants dominate.
